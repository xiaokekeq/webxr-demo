# 基于移动端 AR 的堤坝三维结构透视与现场辅助核查技术路线

## 1. 项目定位

本项目面向堤坝巡查、堤防结构核查和风险辅助识别场景，利用移动端 AR 技术，将堤坝三维结构模型、监测点、风险区域和现场核查信息叠加到真实环境中。

系统目标不是简单地把模型显示在摄像头前，而是：

> 将堤坝三维模型稳定锁定到真实堤坝空间位置中，使巡查人员在现场移动时，可以从不同角度观察堤顶、坝体、迎水坡、背水坡、坝基、防渗结构、排水结构和风险区域。

---

## 2. 当前基础

目前系统已经具备以下能力：

1. **粗配准**

   * 通过 GPS / 经纬度 / IMU 姿态，将堤坝三维模型初步放置到现场附近。
   * 粗配准解决“模型大概在哪里”的问题。

2. **人工微调**

   * 支持对模型进行平移、旋转、缩放等交互调整。
   * 微调用于修正粗配准后的局部偏差。

3. **AR 叠加显示**

   * 已实现堤坝模型在移动端 AR 场景中的加载与显示。
   * 已具备基础图层显示、隐藏、透明叠加等交互能力。

当前主要问题是：

> 模型虽然可以放置到现场附近，但缺少基于现场控制点的精确配准机制，因此在堤顶线、坡脚线、涵闸位置、监测点位置等关键部位仍可能出现错位、漂浮和空间不稳定。

## 5. 坐标系统设计

系统中至少包含四类坐标：

| 坐标系统        | 作用                          |
| ----------- | --------------------------- |
| WGS84 经纬度坐标 | 表示堤坝在真实地理空间中的位置             |
| ENU 局部坐标    | 将经纬度转换为以米为单位的局部坐标           |
| 模型局部坐标      | 堤坝三维模型自身坐标                  |
| AR 世界坐标     | WebXR / Three.js 中实际渲染使用的坐标 |

推荐转换链路：

```text
WGS84 经纬度 / 高程
  ↓
ECEF 地心坐标
  ↓
ENU 局部东北天坐标
  ↓
Three.js 世界坐标
  ↓
AR 世界坐标
```

Three.js 中建议约定：

```text
X 轴 = East，向东
Y 轴 = Up，向上
Z 轴 = -North，向北的反方向
```

这样可以让真实堤坝位置、三维模型和 AR 场景保持统一。

## 6. 粗配准技术路线

粗配准用于将堤坝模型快速放到用户当前位置附近。

### 6.1 输入数据

```text
用户当前位置：
- latitude
- longitude
- altitude

堤坝模型基准点：
- modelOriginLatitude
- modelOriginLongitude
- modelOriginAltitude

设备姿态：
- compass heading
- IMU attitude
- WebXR camera pose
```

### 6.2 处理流程

```text
1. 获取用户 GPS 位置
2. 获取堤坝模型基准点位置
3. 将两者转换到 ENU 局部坐标
4. 计算堤坝模型相对于用户的初始偏移
5. 根据设备朝向修正模型初始旋转
6. 将堤坝模型放入 AR 世界坐标
```

### 6.3 作用边界

粗配准只能解决初始定位，不能作为最终精确结果。

普通移动端 GPS 误差通常达到米级，而堤坝现场核查需要关注堤顶线、坡脚线、涵闸、监测点等局部位置，因此必须继续进行：

```text
人工微调
控制点精配准
模型世界锁定
```

## 7. AR 贴地与模型稳定放置

堤坝模型叠加时，最容易出现的问题是“漂浮感”。因此模型放置时应优先保证贴地和空间稳定。

### 7.1 高度处理原则

```text
水平位置 X / Z 可以来自粗配准或微调
高度 Y 应优先来自 WebXR hit-test 地面
```

不建议直接使用 GPS 高程作为模型高度，因为 GPS 高程误差较大，容易导致模型悬空或插入地面。

### 7.2 模型底面对齐

模型放置时不能简单地让模型原点等于地面点，因为很多模型的原点位于模型中心，而不是底部。

正确逻辑是：

```text
堤坝模型外包盒底部 = hit-test 命中的地面高度
```

示例：

```ts
const wrapper = new THREE.Group();
wrapper.add(model);

model.updateMatrixWorld(true);

const box = new THREE.Box3().setFromObject(model);
const bottomY = box.min.y;

// 将模型底部移动到 wrapper 的 y = 0 平面
model.position.y -= bottomY;

// wrapper 的位置对应真实地面放置点
wrapper.position.copy(hitPosition);
```

这样可以有效减少堤坝模型悬空、插地或漂浮的问题。

## 8. 人工微调技术路线

人工微调用于修正粗配准后的局部偏差。

支持操作包括：

```text
前后移动
左右移动
上下微调
水平旋转
整体缩放
重置
确认锁定
```

建议维护独立的微调矩阵，而不是直接破坏模型原始坐标：

```text
T_final = T_ar_from_enu_coarse × T_manual_adjust
```

加入控制点精配准后，矩阵组合变为：

```text
T_final = T_ar_from_enu_coarse × T_control_refine × T_manual_adjust
```

## 9. 控制点精配准技术路线

控制点精配准是下一阶段核心。

它的目标是：

> 利用现场可识别的真实点，将堤坝模型坐标与真实 AR 空间坐标精确对应。

---

### 9.1 堤坝场景中的控制点类型

堤坝项目中，控制点不应再按管线井盖、地下管线等对象设计，而应优先选择堤坝现场稳定、易识别的结构点。

推荐控制点包括：

```text
堤顶边线点
堤顶道路边缘点
迎水坡坡肩点
背水坡坡肩点
坡脚点
涵闸角点
出水口角点
水尺位置
监测桩位置
渗压监测点位置
界桩 / 里程桩
明显标识牌位置
路面黄线角点
现场二维码 / Marker 点
```

控制点必须同时具备两类坐标：

| 坐标      | 含义                  |
| ------- | ------------------- |
| 模型控制点坐标 | 该点在堤坝模型或 ENU 坐标中的位置 |
| 现场控制点坐标 | 用户在 AR 现场采集到的真实位置   |

---

### 9.2 控制点采集流程

推荐交互流程：

```text
1. 用户进入“精配准”模式
2. 系统显示堤坝模型中的控制点列表
3. 用户选择一个模型控制点
4. 用户移动手机，将屏幕准星对准现实中的对应点
5. 系统通过 hit-test 获取该现实点的 AR 世界坐标
6. 用户点击“采集控制点”
7. 重复采集 2 到 3 个以上控制点
8. 系统计算精配准矩阵
9. 应用到堤坝模型根节点
10. 显示 RMS 误差
11. 用户确认并锁定模型
```

---

### 9.3 控制点数量与修正能力

| 控制点数量   | 可修正内容                   | 适用情况                |
| ------- | ----------------------- | ------------------- |
| 1 个点    | 平移                      | 只能将模型某一点拉到现实对应点     |
| 2 个点    | 平移 + 水平旋转 + 可选尺度        | 适合堤顶线、坡脚线、道路边线等线性结构 |
| 3 个及以上点 | 平移 + 旋转 + 尺度 + RMS 误差评估 | 推荐方案，稳定性更高          |

堤坝属于典型线性工程，现场很多结构具有明显方向性，因此两点配准可以先作为可用版本。

---

## 10. 两点水平配准方案

堤坝现场中，可以优先选择两个沿堤坝方向分布的控制点，例如：

```text
两个堤顶边线点
两个坡脚点
两个里程桩
涵闸两侧角点
道路黄线上的两个点
```

设：

```text
模型点：M1, M2
现实 AR 点：A1, A2
```

计算：

```text
模型方向向量：Vm = normalize(M2 - M1)
现实方向向量：Va = normalize(A2 - A1)

水平旋转角：angle = angleBetween(Vm.xz, Va.xz)
尺度：scale = length(A2 - A1) / length(M2 - M1)
平移：translation = A1 - transform(M1)
```

两点配准适合先解决：

```text
堤坝模型方向不对
堤顶线与现实边线不重合
模型整体偏移
模型比例轻微不一致
```

---

## 11. 三点及以上精配准方案

当控制点数量达到 3 个及以上时，推荐使用相似变换配准。

目标是求解：

```text
A ≈ s × R × M + t
```

其中：

```text
M：堤坝模型控制点坐标
A：现场 AR 控制点坐标
s：尺度
R：旋转
t：平移
```

处理步骤：

```text
1. 计算模型控制点质心
2. 计算现场控制点质心
3. 两组点分别去中心化
4. 计算协方差矩阵
5. 通过 Horn / Kabsch / SVD 求解旋转
6. 计算尺度
7. 计算平移
8. 得到控制点精配准矩阵
9. 计算 RMS 误差
```

RMS 误差可用于判断配准质量：

```text
RMS < 0.10 m：较好
0.10 m <= RMS < 0.30 m：可用
0.30 m <= RMS < 1.00 m：偏差明显，需要重新采点
RMS >= 1.00 m：不建议使用
```

## 12. 世界锁定策略

控制点精配准完成后，模型不应继续每帧跟随 GPS / IMU 更新。

正确逻辑是：

```text
粗配准只用于初始化
微调用于人工修正
控制点用于精确对齐
配准完成后锁定堤坝模型根节点
用户移动时只更新 AR 相机
```

核心代码逻辑：

```ts
modelRoot.matrixAutoUpdate = false;
modelRoot.matrix.copy(finalRegistrationMatrix);
modelRoot.updateMatrixWorld(true);
```

渲染循环中只更新相机、标签和图层：

```ts
renderer.setAnimationLoop((time, frame) => {
  updateLabels(camera);
  updateClipping(camera);
  updateLayerVisibility();
  renderer.render(scene, camera);
});
```

不要在渲染循环中继续执行：

```ts
modelRoot.position.copy(gpsPosition);
modelRoot.rotation.copy(imuRotation);
```

否则用户走动时，模型会出现漂移、抖动或跟随相机滑动的问题。

## 15. 标签跟随

堤坝标签应绑定真实三维对象，例如：

```text
堤顶

坝体
坝基
反滤层

涵闸
水尺
渗压监测点

```

每一帧将三维坐标投影到屏幕：

```ts
function updateLabel(
  labelEl: HTMLElement,
  worldPoint: THREE.Vector3,
  camera: THREE.Camera
) {
  const p = worldPoint.clone().project(camera);

  const x = (p.x * 0.5 + 0.5) * window.innerWidth;
  const y = (-p.y * 0.5 + 0.5) * window.innerHeight;

  labelEl.style.transform = `translate(${x}px, ${y}px)`;
}
```

这样用户移动时，标签会自然跟随对应堤坝结构变化。

## 18. 最终矩阵组合

堤坝模型最终变换矩阵建议组织为：

```text
T_final =
T_ar_from_enu_coarse
× T_control_refine
× T_manual_adjust
× T_model_offset
```

含义：

| 矩阵                   | 作用               |
| -------------------- | ---------------- |
| T_ar_from_enu_coarse | GPS / IMU 粗配准    |
| T_control_refine     | 控制点精配准           |
| T_manual_adjust      | 人工微调             |
| T_model_offset       | 模型原点、底面对齐、单位缩放修正 |

最终应用到堤坝模型根节点：

```ts
modelRoot.matrixAutoUpdate = false;
modelRoot.matrix.copy(T_final);
modelRoot.updateMatrixWorld(true);
```

## 20. 当前优先级

当前最应该优先做：

```text
1. 修正堤坝模型贴地问题
2. 确认模型放置后不再每帧跟随 GPS / IMU
3. 加入堤坝控制点采集流程
4. 先实现两点水平配准
5. 再实现三点以上 RMS 精配准
6. 最后叠加透明、剖切、标签和风险区
```

当前不建议优先做：

```text
复杂大屏 UI
完整真实遮挡
多人协同
云端同步
高精度 RTK 接入
复杂数字孪生平台
```

这些功能可以作为后续扩展，不应影响当前核心配准链路的完成。

## 21. 结论

本项目当前已经具备粗配准、人工微调和 AR 模型显示基础。

下一步技术重点应从“能显示堤坝模型”转向“堤坝模型与真实现场精确对齐”。

最终技术路线可以概括为：

```text
粗配准负责带到堤坝现场附近
人工微调负责临时修正
控制点配准负责精确对齐
世界锁定负责行走时稳定观察
结构透视负责展示堤坝内部结构
风险与数据叠加负责辅助现场核查
```

最终实现效果应为：

> 巡查人员在堤坝现场移动时，堤坝三维模型稳定固定在真实空间中，可以通过 AR 查看坝体结构、坝基、防渗与排水结构、涵闸、监测点和风险区域，从而辅助完成堤坝现场核查。
